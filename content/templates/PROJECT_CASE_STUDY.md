# 프로젝트 서브 페이지 작성 가이드 (포트폴리오용)

콘텐츠 저장소의 **프로젝트별 폴더** 아래 아키텍처·프론트·백엔드용 마크다운은 **사내 위키가 아니라 이직·채용용 케이스 스터디**로 작성합니다.

## 원칙

1. **문제 우선**: 기능·엔드포인트 나열이 아니라 “무엇이 어려웠고 왜 중요했는지”부터 씁니다.
2. **결정 근거**: 기술 A를 쓴 이유, 대안 B·C와의 트레이드오프를 한두 문장이라도 적습니다.
3. **성과**: 가능하면 수치·정성적 효과(유지보수, 확장, 안정성)를 적습니다.
4. **간결**: 챌린지당 짧은 단락 위주. API 전체 목록·스토어 전체 목록은 피합니다.
5. **시각**: 페이지당 Mermaid **0~2개**만. 고수준 흐름에만 사용합니다.

## 권장 frontmatter

각 파일 상단 YAML은 기존과 동일하게 제목·슬러그·카테고리·정렬 순서 필드를 유지합니다.

## 본문 구조 (복붙용)

```markdown
## 핵심 기술 (한 줄 요약)
이 탭에서 다루는 스택과 역할을 1~2문장으로.

## 기술적 도전과 해결

### Challenge: (짧고 구체적인 제목)

**상황** — 비즈니스·제품 맥락.

**문제** — 기술적·운영적 난점.

**접근** — 검토한 대안과 선택 이유.

**해결** — 핵심 아이디어·패턴 (필요 시 코드/다이어그램 1개).

**성과** — 정량 또는 정성적 결과.

(챌린지 2~4개 반복)

## 설계 메모 (선택)
추가로 남기고 싶은 트레이드오프·한계·다음 개선만 bullet로.
```

## 피할 것

- REST 엔드포인트 전체 테이블, 클래스·파일 목록만 나열
- 동일 내용을 아키텍처·FE·BE 탭에 세 번 복붙
- Mermaid만으로 페이지를 채우기

## 참고

- 개요·갤러리·하이라이트는 **프로젝트 루트 개요 문서**의 메타데이터와 본문에서 다룹니다.
- 상세 구현 문서가 필요하면 별도 저장소·위키에 두고, 포트폴리오에는 요약만 링크합니다.
