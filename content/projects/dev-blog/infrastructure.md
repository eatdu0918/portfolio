## 핵심 기술 (한 줄 요약)

프로덕션은 **Vercel**에 올려 Git 푸시 기반으로 빌드·배포하며, DB는 **PostgreSQL(호스팅 연결 풀 권장)**입니다.

## 기술적 도전과 해결

### Challenge: App Router·Server Actions를 “서버리스 친화”로

**상황** — 댓글 같은 동적 로직을 별도 컨테이너 없이 돌리고 싶었습니다.

**문제** — 함수 콜드 스타트·DB 커넥션 한도가 걸립니다.

**접근** — 정적 세그먼트는 **빌드 시 경로 목록을 미리 생성**하는 방식으로 넓히고, DB 접근은 **짧은 트랜잭션**으로 유지했습니다.

**해결** — Prisma 클라이언트 생성 패턴을 서버리스 가이드에 맞췄습니다.

**성과** — **호스팅 제약 안에서 풀스택**을 유지했습니다.

### Challenge: 시크릿·환경 변수 관리

**상황** — DB URL이 저장소에 올라가면 즉시 사고입니다.

**문제** — 팀원마다 다른 로컬 값이면 재현이 어렵습니다.

**접근** — Vercel 프로젝트 시크릿과 로컬 **환경 변수 파일**을 **동일 키 스키마**로 맞췄습니다.

**해결** — README 수준의 최소 문서로 onboarding 비용을 줄였습니다.

**성과** — PR마다 “환경 때문에 실패” 비율을 줄였습니다.

### Challenge: 관측·확장 여지

**상황** — 개인 프로젝트라도 운영 로그는 필요합니다.

**문제** — 커스텀 APM까지는 과합니다.

**접근** — Vercel 빌드/함수 로그로 **1차 관측**을 하고, 트래픽이 커지면 캐시 헤더·DB 플랜을 단계적으로 조정할 여지를 남겼습니다.

**해결** — HTTPS·기본 도메인을 그대로 활용했습니다.

**성과** — **비용 대비 운영 투명성**을 확보했습니다.

## 설계 메모

- 라이브 URL은 **프로젝트 개요 문서의 메타데이터에 적힌 데모 주소 필드**로 노출합니다. 터널/도메인이 바뀌면 그 값만 갱신하면 됩니다.
