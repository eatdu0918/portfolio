## 핵심 기술 (한 줄 요약)

**Main: Vue 3 + PrimeVue + Cesium**, **Admin/Edge: Quasar**, 상태는 **Pinia**, 실시간은 **SockJS/STOMP**, 현장 영상은 **HLS.js**입니다. 좌표는 **MGRS**로 사용자에게 일관 표시합니다.

## 기술적 도전과 해결

### 도전 과제 1: Cesium 3D 엔진 기반의 입체적 전장 정보 시각화 및 통합

**상황** — 부대 편제부호, 실시간 기동로, 방어선 및 AI 분석 결과(래스터 오버레이)를 하나의 3D 공간 상에 일관된 타임라인으로 겹쳐서 보여주어야 했습니다.

**문제** — 다양한 객체 타입(Billboard, Polyline, Polygon 등)과 대량의 실시간 데이터를 처리하는 과정에서 렌더링 로직이 복잡해지고, 프레임 드랍(Frame Drop)이나 상태 동기화 오류가 발생할 가능성이 높았습니다.

**접근** — **3D 뷰어 전용 인터페이스 계층(Viewer Interface Layer)**을 설계했습니다. 핵심 데이터 스토어는 상태 관리(Pinia)에 집중하고, 렌더링 엔진은 해당 상태를 구독하여 상황 변화에 따라 객체의 생명주기를 관리하도록 역할을 분리했습니다.

**해결** — WebSocket으로 유입되는 실시간 상황 전파 데이터와 사용자의 인터랙션을 이벤트 허브로 통합하여, 3D 씬(Scene)의 최신성을 보장하는 단방향 데이터 흐름을 구축했습니다.

**성과** — 수천 개의 군사 객체가 포함된 복잡한 전장 상황에서도 **초당 60프레임(60 FPS) 이상의 안정적인 성능**을 확보했으며, 지휘관에게 직관적인 3D 작전 뷰를 제공하게 되었습니다.

### 도전 과제 2: 야전 환경(Edge)의 실시간 영상 스트리밍 안정성 확보

**상황** — 드론 및 외부 정찰 장비에서 송출되는 HLS 라이브 영상을 저지연으로 재생하고, 정찰 임무의 생명주기에 맞춰 자원을 관리해야 했습니다.

**문제** — 모바일 및 엣지 장비의 브라우저 특성에 따라 영상 끊김이 발생하거나, 재생 중인 인스턴스가 올바르게 해제되지 않아 발생하는 메모리 누수 문제가 발생했습니다.

**접근** — **명시적 라이브 스트림 생명주기 관리자**를 구현했습니다. 정찰 세션의 시작과 종료 이벤트에 맞춰 플레이어 인스턴스를 동적으로 할당하고 해제하는 로직을 적용했습니다.

**해결** — 브라우저별 HLS 지원 여부에 따라 네이티브 재생과 라이브러리(hls.js) 재생을 자동으로 선택하는 폴리필(Polyfill) 구조를 도입하여 호환성을 극대화했습니다.

**성과** — 열악한 야전 네트워크 환경에서도 **영상 수신 성공률을 95% 이상**으로 높였으며, 장시간 운용 시에도 시스템 자원 점유율을 일정하게 유지하는 데 성공했습니다.

### 도전 과제 3: 방대한 운영 데이터(Master Data) 관리 효율화 및 표준화

**상황** — 부대, 무기, 국면, 환경 설정 등 수십 종의 다양한 마스터 데이터를 관리하기 위한 어드민 페이지가 대량으로 필요했습니다.

**문제** — 각 페이지마다 독립적으로 테이블과 폼(Form)을 개발할 경우 디자인 일관성이 떨어지고 유사 코드의 중복으로 인해 유지보수 비용이 급증했습니다.

**접근** — **선언적 UI 패턴 기반의 공통 컴포넌트**를 개발했습니다. 데이터 필드 정의만으로 목록 조회, 필터링, 편집 모달이 자동으로 생성되는 표준 템플릿을 구축했습니다.

**해결** — 프레임워크(Quasar)의 유효성 검사 규칙과 서버 DTO 명세를 일치시켜, 프론트엔드와 백엔드 간의 데이터 정합성을 컴파일 타임에 가깝게 보장하도록 설계했습니다.

**성과** — 신규 마스터 엔티티 추가 시 **개발 공수를 기존 대비 50% 이상 절감**했으며, 일관된 UI/UX 제공을 통해 운영자의 학습 비용을 낮추었습니다.

### Challenge: HTTP 클라이언트 401/403과 JWT 만료 UX

**상황** — 세 클라이언트 모두 동일한 토큰 규칙을 써야 했습니다.

**문제** — 만료 시 무한 재시도나 빈 화면이 나면 현장 신뢰가 떨어집니다.

**접근** — **요청 가로채기**에서 **401 → 로그인 라우트**, **403 → 명시적 에러 처리**로 단순화했습니다.

**해결** — 공통 모듈에 헤더 주입을 두어 세 앱이 같은 규칙을 공유했습니다.

**성과** — 운용 중 **인증 이슈 재현·대응**이 쉬워졌습니다.

## 설계 메모

- MGRS는 프론트·백 각각 **동일 규칙의 좌표 라이브러리**로 변환해 **표시와 저장이 어긋나지 않게** 맞췄습니다.
