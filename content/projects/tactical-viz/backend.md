## 핵심 기술 (한 줄 요약)

**Spring Boot 3.3 · Java 17 · MyBatis · MapStruct · Spring Security(JWT)** 멀티모듈이며, **WebSocket STOMP**, **Ehcache**, **GeoTools/JTS** 좌표 처리, 외부 **AI 모델 HTTP 콜백**을 사용합니다.

## 기술적 도전과 해결

### 도전 과제 1: 멀티모듈 아키텍처를 통한 도메인 비즈니스 로직의 통합 관리

**상황** — 지휘용 작전 API와 관리 백오피스용 API는 보안 요구사항에 따라 배포 단위와 포트가 완전히 분리되어야 하지만, 부대 정보나 작전 코드와 같은 핵심 도메인 모델은 공유되어야 합니다.

**문제** — 물리적으로 서비스를 나누기 위해 코드를 복사(Copy & Paste)할 경우, 요구사항 변경 시 두 서비스에 동시에 반영하기 어렵고 휴먼 에러로 인한 데이터 불일치가 발생할 우려가 컸습니다.

**접근** — **Maven 기반 멀티모듈 아키텍처**를 구축했습니다. 공통 라이브러리 모듈에 MyBatis Mapper XML, 데이터 접근 계층(Repository), 공유 DTO 및 보안 기본 설정을 집약시키고 각 서비스(지휘/관리)는 최소한의 진입점만 유지하도록 설계했습니다.

**해결** — 한 번의 수정으로 두 서비스에 비즈니스 논리가 동시에 반영되는 상속 구조를 완성했으며, CI/CD 파이프라인에서 각 모듈을 독립적인 이미지로 빌드하여 배포 효율을 높였습니다.

**성과** — 중복 코드를 **80% 이상 제거**하여 유지보수 비용을 획기적으로 낮췄으며, 데이터 모델의 일관성을 확보하여 시스템 전반의 신뢰도를 높였습니다.

### 도전 과제 2: 이기종 AI 분석 모델과의 효율적인 데이터 직렬화 및 연계

**상황** — 외부에서 연동되는 AI 분석 모델들이 각기 다른 데이터 형식(예: 이동로는 WKB, 타격범위는 Base64 GeoTIFF)을 반환하여 시스템 통합의 복잡도가 높았습니다.

**문제** — 가공되지 않은 데이터를 클라이언트(Cesium 3D)에 그대로 전달할 경우 프론트엔드에서 파싱 부하가 발생하고, 좌표계 불일치로 인한 시각화 오류가 빈번했습니다.

**접근** — 백엔드 레이어에서 **데이터 정규화(Normalization) 파이프라인**을 구현했습니다. AI 모델의 응답을 수신하는 즉시 필요한 좌표계 변환과 GeoJSON/WKB 파싱을 완료하여, 클라이언트는 즉시 렌더링 가능한 형태의 데이터만 수신하도록 구조화했습니다.

**해결** — 비동기 작업의 추적을 위해 **세션 키 기반의 콜백 계약**을 명시하고, 메시지 브로커를 거치지 않고도 신속하게 결과를 전파할 수 있는 실시간 처리 통로를 구축했습니다.

**성과** — 프론트엔드의 비즈니스 로직 복잡도를 대폭 낮추었으며, 초동 조치 시 필수적인 분석 결과의 시각화 속도를 크게 개선했습니다.

### 도전 과제 3: 실시간 정보 전파의 보안성 및 트래픽 최적화

**상황** — GPS 정보와 같은 상황 공유 데이터는 모든 접속자가 봐야 하지만, 특정 AI 분석 결과는 요청한 지휘관 개인에게만 전달되어야 하는 보안 요구사항이 있었습니다.

**문제** — 단일 WebSocket 채널에 모든 메시지를 브로드캐스트할 경우 불필요한 네트워크 트래픽이 발생하고 민감한 정보가 노출될 위험이 있었습니다.

**접근** — **메시지 유형별 채널 세분화 전략**을 도입했습니다. 전역 상황 공유를 위한 공용 토픽(Public Topic)과 개별 요청 처리를 위한 전용 큐(Private Queue)를 분리하여 운영했습니다.

**해결** — WebSocket 연결 시 JWT 인증 필터를 통해 구독 권한을 실시간으로 확인하고, 인가된 사용자만 특정 채널에 접속할 수 있도록 Spring Security와 STOMP 인터셉터를 연동했습니다.

**성과** — 실시간 메시징 환경에서도 **강력한 데이터 인가 체계**를 구축했으며, 클라이언트별 맞춤형 정보 전달을 통해 네트워크 효율을 최적화했습니다.

### Challenge: 반복 조회 부하(Ehcache)

**상황** — 공통코드·부대 목록 등 읽기 쿼리가 매우 잦습니다.

**문제** — DB 커넥션이 병목이 됩니다.

**접근** — **Ehcache 3**로 열역학적으로 자주 읽히는 데이터를 캐시했습니다.

**해결** — 캐시 키·TTL을 운영 피드백에 맞춰 조정할 수 있게 했습니다.

**성과** — 피크 시간 **응답 지연을 완화**했습니다.

## 설계 메모

- REST 계약은 **SpringDoc/OpenAPI**로 노출하고, 컨트롤러–도메인 경계는 **MapStruct 매핑 인터페이스**로 고정해 변경 시 컴파일 타임에 어긋남을 잡도록 했습니다. 세부 엔드포인트 목록은 API 문서 산출물을 기준으로 유지합니다.
