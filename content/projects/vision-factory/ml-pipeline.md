## 핵심 기술 (한 줄 요약)

**RabbitMQ로 단계 분리된 Python 워커**(FFmpeg 프레임 추출, YOLOv8+BYTETracker, 블러/합성, 제로샷 트래킹, 좌표 기반 블러)와 **ONNX Runtime·TensorRT** 등 GPU 경로를 조합했습니다.

## 기술적 도전과 해결

### 도전 과제 1: 비디오 디코딩과 AI 추론 프로세스의 물리적 분리 및 자원 최적화

**상황** — 다양한 코덱과 해상도를 가진 원본 영상을 AI 모델이 처리할 수 있는 일관된 형태의 프레임 시퀀스로 변환하는 전처리가 필요합니다.

**문제** — 단일 프로세스에서 영상 디코딩(CPU 부하)과 AI 추론(GPU 부하)을 동시에 처리할 경우, 시스템 자원 예측이 어렵고 특정 단계의 병목이 전체 파이프라인의 성능 저하로 이어졌습니다.

**접근** — FFmpeg 기반의 **프레임 추출 전용 워커(CPU 집중)**와 YOLO/TensorRT 기반의 **객체 탐지 전용 워커(GPU 집중)**를 RabbitMQ 큐를 통해 물리적으로 분리했습니다.

**해결** — 각 워커의 특성에 맞는 하드웨어 가속(ONNX Runtime, TensorRT)을 개별적으로 적용하고, 단계별 처리량을 독립적으로 확장(Scaling)할 수 있는 구조를 구축했습니다.

**성과** — 디코딩과 추론 부하를 분리함으로써 시스템 안정성을 확보했으며, 병목 구간에만 자원을 추가 할당하여 전체 처리 성능을 40% 이상 개선했습니다.

### 도전 과제 2: 블러 처리와 생성형 합성 모드 간의 품질 및 비용 최적화

**상황** — 보안 규정에 따라 단순 블러(Blur) 처리만으로 충분한 경우와, 자연스러운 영상 품질을 위해 생성 모델을 이용한 얼굴 합성(Replace)이 필요한 경우가 공존합니다.

**문제** — 모든 작업에 생성형 모델을 적용하면 비용과 시간이 과다하게 소요되고, 단순 블러만 제공하면 고품질 영상 제작이라는 고객 니즈를 충족하기 어려웠습니다.

**접근** — 작업 요청 시 인자로 모드를 분리하여 **경량 블러 워커**와 **고성능 합성 워커(Stable Diffusion + ControlNet)**가 선택적으로 작업을 소비하도록 설계했습니다.

**해결** — 동일한 탐지 결과(BBox 좌표)를 공유하되 후속 처리만 다르게 수행하는 분기 로직을 파이프라인에 이식했습니다.

**성과** — 고객의 서비스 수준 협약(SLA)에 따라 비용 중심과 품질 중심의 옵션을 탄탄하게 제공할 수 있게 되었습니다.

### 도전 과제 3: 제로샷 트래킹과 CPU 기반 후처리를 통한 GPU 경쟁 완화

**상황** — 사용자가 수동으로 선택한 객체를 영상 전 구간에서 추적하여 비식별화하는 기능이 필요합니다.

**문제** — 수동 보정 단계까지 고성능 GPU 연산을 강제하면 인프라 운영 비용이 급격히 상승합니다.

**접근** — 트래킹 연산은 **제로샷 트래킹 워커**가 빠르게 좌표 궤적을 산출하고, 실제 영상 렌더링(블러 합성)은 OpenCV 중심의 **CPU 최적화 워커**가 처리하도록 역할을 분담했습니다.

**해결** — 트래킹 결과물을 표준 JSON 스키마로 정의하여 렌더링 워커가 파이프라인의 마지막 단계에서 자원을 효율적으로 사용하게 했습니다.

**성과** — 수동 작업의 실시간 응답성은 유지하면서도 전체 GPU 사용률을 낮추어 운영 비용을 최적화했습니다.

## GPU/CPU 배치 원칙

| 단계 | 자원 초점 | 이유 |
|------|------------|------|
| 프레임 추출 | CPU | FFmpeg 디코딩 |
| 탐지·트래킹 | GPU | 딥러닝 추론·TensorRT |
| 합성 교체 | GPU | 확산 모델 |
| 좌표 블러 | CPU | 궤적 적용 중심 |

## 설계 메모

- 워커 이미지는 CUDA 기반으로 컨테이너화해 **같은 큐를 여러 인스턴스가 나눠 소비**하게 했습니다. 배포 시 주입 설정으로 브로커·Redis만 바꿔 이중화 연습이 가능합니다.
