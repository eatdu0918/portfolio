## 핵심 기술 (한 줄 요약)

**Spring Boot 3.2 · Maven 멀티모듈(사용자 API·관리 백오피스·공통 모듈) · MariaDB · MyBatis(+일부 JPA) · Spring Security(JWT) · RabbitMQ · Redis · 실시간 웹소켓 메시징**입니다.

## 기술적 도전과 해결

### 도전 과제 1: 사용자 API와 백오피스 API의 배포 및 보안 계층 분리

**상황** — 운영 효율화를 위한 백오피스 기능과 일반 사용자의 실시간 영상 처리 기능은 보안 요구사항과 트래픽 패턴이 완전히 다릅니다.

**문제** — 단일 애플리케이션 구조에서는 사용자 트래픽 급증 시 관리 기능의 안정성이 저하될 수 있으며, 권한 설정이 복잡해져 보안 취약점이 발생할 우려가 있었습니다.

**접근** — **Maven 멀티모듈 아키텍처**를 활용하여 사용자용 API와 관리자용 백오피스를 별도의 실행 파일(JAR)로 분리하고, 비즈니스 로직과 데이터 접근 계층은 공통 모듈로 관리했습니다.

**해결** — 각 서비스의 역할에 맞는 별도의 보안 필터 체인과 포트를 구성하여 공격 노출면을 최소화하고 리소스 할당을 최적화했습니다.

**성과** — 서비스별 독립적인 스케일링이 가능해졌으며, 마이크로서비스로의 전환이 용이한 탄탄한 기반을 마련했습니다.

### 도전 과제 2: SaaS 테넌트 격리 및 통합 인증 체계 구축

**상황** — 다수의 기업 고객이 동시에 사용하는 SaaS 환경에서는 각 고객사(테넌트) 간의 데이터가 철저히 격리되어야 합니다.

**문제** — 인증 토큰에 테넌트 식별 정보가 충분하지 않을 경우, 데이터 조회나 작업 큐 할당 시 다른 고객사의 정보가 유출될 위험이 있었습니다.

**접근** — **JWT(JSON Web Token)** 클레임에 사용자 식별자뿐만 아니라 회사 코드, 전용 작업 큐 힌트를 포함하여 인증과 인가를 통합 관리했습니다.

**해결** — Spring Security의 필터 체인을 통해 모든 요청에서 테넌트 유효성을 강제하고, DB 접근 시에도 테넌트 식별자를 필수로 포함하도록 설계했습니다.

**성과** — 복잡한 멀티테넌트 환경에서도 안정적인 데이터 격리를 구현했으며, 운영 이슈 발생 시 신속한 추적이 가능해졌습니다.

### 도전 과제 3: 비동기 파이프라인과 실시간 상태 동기화 설계

**상황** — AI 워커에서 처리되는 수 분 이상의 초장기 작업 상태를 사용자에게 실시간으로 전달해야 합니다.

**문제** — 모든 하위 이벤트를 데이터베이스에 직접 기록하고 폴링(Polling) 방식으로 조회할 경우, DB 부하가 급증하고 실시간성이 떨어지는 문제가 있었습니다.

**접근** — **RabbitMQ를 통한 작업 위임, Redis를 이용한 상태 일시 저장, Redis Pub/Sub을 통한 이벤트 전파** 구조를 설계했습니다.

**해결** — 백엔드 API가 Redis 채널을 구독하고 있다가 결과가 유입되면 웹소켓(STOMP)을 통해 브라우저에 즉시 푸시하는 방식으로 구현했습니다.

**성과** — DB 부하를 최소화하면서도 1초 내외의 지연 시간으로 처리 상태를 공유할 수 있는 고성능 알림 시스템을 구축했습니다.
