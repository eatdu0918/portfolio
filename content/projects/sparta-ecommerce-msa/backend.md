## 핵심 기술 (한 줄 요약)

**Spring Boot 3.2 · Java 17** 마이크로서비스, **Spring Cloud Gateway(WebFlux)**, **OpenFeign**, **Kafka**, **Redis**, **서비스별 MySQL**, **Actuator/Prometheus**, **Logstash Logback Encoder**, **TestContainers**입니다.

## 기술적 도전과 해결

### Challenge: 게이트웨이에 인증·횡단 관심사를 모으기

**상황** — 각 서비스에 JWT 검증을 복제하고 싶지 않았습니다.

**문제** — 게이트웨이가 비대해지면 배포 리스크가 커집니다.

**접근** — **라우팅 + JWT + Redis(세션/블랙리스트/레이트 제한 등 용도)**를 게이트웨이 필터 체인으로 모았습니다.

**해결** — 비즈니스 규칙은 여전히 도메인 서비스에 남겼습니다.

**성과** — 클라이언트는 **한 진입점**만 알면 되고, 보안 정책 변경이 한곳에 모입니다.

### Challenge: 동기(Feign)와 비동기(Kafka) 혼합

**상황** — 주문 생성 직후 할인·재고 확인은 빠른 응답이 필요합니다.

**문제** — 이벤트만으로는 사용자가 기다리는 답을 주기 어렵습니다.

**접근** — 즉시 응답이 필요한 조합은 **Feign**, 후속 알림·보상은 **Kafka**로 분리했습니다.

**해결** — 실패 시 Saga/보상 패턴을 학습 범위로 명시했습니다.

**성과** — MSA를 설계할 때의 **동기/비동기 타협점을 개인 레퍼런스**로 축적했다.

### Challenge: 분산 환경 관측·로그

**상황** — 서비스가 늘면 원인 추적이 어렵습니다.

**문제** — 콘솔 로그만으로는 부족합니다.

**접근** — Actuator 메트릭을 Prometheus가 스크랩할 수 있게 하고, Logback encoder로 **구조화 로그**를 남겼습니다.

**해결** — 게이트웨이와 핵심 서비스부터 메트릭을 통일했습니다.

**성과** — 장애 대응 시 **“어느 hop에서 느려졌는지”**를 좁히기 쉬워졌습니다.

### Challenge: CI에서도 같은 전제로 통합 테스트

**상황** — 로컬에서만 되는 테스트는 신뢰가 없습니다.

**문제** — MySQL·Kafka를 직접 띄우기엔 CI가 무겁습니다.

**접근** — **TestContainers**로 외부 의존성을 띄워 통합 테스트를 작성했습니다.

**해결** — 파이프라인에서 동일 스위트를 돌립니다.

**성과** — 리팩터링 시 **회귀 비용**을 줄였습니다.

## 설계 메모

- 구체 토픽/컨슈머 그룹 설계는 진행 중일 수 있어, 포트폴리오에는 **패턴과 학습 목표**만 남깁니다.
