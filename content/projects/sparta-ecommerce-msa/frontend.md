## React + Vite SPA

이커머스 사용자 화면은 **React 19**와 **Vite 7** 기반의 단일 페이지 애플리케이션으로 구성했습니다. API는 게이트웨이 단일 진입점을 바라보도록 두어, 브라우저는 서비스별 호스트를 알 필요 없이 호출합니다.

### 기술 스택

| 분류 | 기술 |
|------|------|
| **런타임·빌드** | React 19, Vite 7, TypeScript |
| **스타일** | Tailwind CSS 4 |
| **서버 상태** | TanStack Query (캐싱·재시도·로딩·에러 상태) |
| **클라이언트 상태** | Zustand (장바구니·UI 등 경량 전역 상태) |
| **결제** | PortOne / Toss Payments SDK 연동 |

### 데이터 흐름

- 목록·상세·주문 등 **REST 응답**은 TanStack Query로 가져오고, 키 설계로 무효화(`invalidate`) 시 관련 화면이 다시 맞춰지도록 했습니다.
- 로그인 이후에는 게이트웨이가 기대하는 방식(예: `Authorization` 헤더)으로 요청을 보내 **JWT 기반 인증** 흐름과 맞췄습니다.

### 결제 UX

결제 단계에서는 외부 PG SDK를 호출하고, 성공·실패·취소 콜백을 처리해 **주문·결제 서버 상태**와 화면 메시지가 어긋나지 않도록 했습니다. (백엔드 이벤트·보상 트랜잭션과의 정합은 백엔드·아키텍처 탭과 연결됩니다.)

로컬 개발 시에는 **ngrok**으로 게이트웨이(또는 백엔드 콜백 엔드포인트)에 공개 URL을 붙여, PG가 요구하는 **리다이렉트·웹훅 URL**을 실제로 호출되게 맞췄습니다. 프론트의 API 베이스 URL도 동일한 터널 호스트를 바라보도록 환경 변수로 전환할 수 있게 두었습니다.

### 배포 관점

클라이언트는 **Blue/Green** 전략을 염두에 두고 빌드·배포 파이프라인과 맞췄습니다. 정적 자산과 API 베이스 URL을 환경별로 분리해 스테이징·프로덕션 전환이 가능하도록 구성했습니다.
