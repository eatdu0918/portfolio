## 핵심 기술 (한 줄 요약)

**React 19 + Vite 7 + TypeScript + Tailwind 4**, 서버 상태는 **비동기 데이터 동기화 라이브러리(React Query 계열)**, 가벼운 전역은 **Zustand**, 결제는 **PortOne / Toss Payments SDK**입니다.

## 기술적 도전과 해결

### Challenge: 여러 서비스 뒤에 숨은 단일 API 베이스

**상황** — 브라우저는 게이트웨이만 알아야 합니다.

**문제** — 서비스별 URL이 새면 CORS·토큰·캐시 키가 모두 깨집니다.

**접근** — 환경 변수로 **API base URL 하나**만 노출하고, 라우팅은 게이트웨이에 위임했습니다.

**해결** — **쿼리 캐시 키**를 리소스 중심으로 설계해 무효화 시 화면이 같이 갱신되게 했습니다.

**성과** — 백엔드 서비스가 늘어도 **클라이언트 설정 변경을 최소화**했습니다.

### Challenge: 결제 리다이렉트·웹훅을 로컬에서 재현

**상황** — PG는 공개 HTTPS 콜백을 요구하는 경우가 많습니다.

**문제** — **로컬 루프백 주소**만으로는 리다이렉트 검증이 어렵습니다.

**접근** — **ngrok**으로 게이트웨이(또는 콜백 대상)에 터널을 열고, 프론트 API 베이스도 같은 호스트로 맞췄습니다.

**해결** — 성공/실패/취소 콜백에서 주문·결제 상태와 UI 메시지가 어긋나지 않게 처리했습니다.

**성과** — “로컬에서만 되는 결제”가 아니라 **외부와 동일한 플로우**를 검증할 수 있었습니다.

### Challenge: 장바구니·UI 상태의 과도한 전역화 방지

**상황** — 서버 상태와 클라이언트 상태를 섞으면 버그가 납니다.

**문제** — 모든 것을 **비동기 캐시 계층**에 넣으면 캐시 무효화가 복잡해집니다.

**접근** — **서버 데이터는 비동기 캐시**, **장바구니 등 UI는 Zustand**로 분리했습니다.

**해결** — 주문 완료 시 관련 항목을 **캐시에서 일괄 무효화**하는 규칙을 문서화했습니다.

**성과** — 상태 디버깅이 **축이 나뉘어** 쉬워졌습니다.

## 설계 메모

- Blue/Green 배포를 염두에 **환경별 자산·API 베이스**를 분리해 두었습니다.
