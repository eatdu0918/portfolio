## 핵심 기술 (한 줄 요약)

**Vue 3 · Vite 7 · Pinia · Tailwind 4 · OpenLayers 10** 위에 **MIL-STD-2525D**(milsymbol / mil-sym-ts-web)·**Vue Flow**·**Turf.js**를 올렸고, 지도 캡처·Excel 보내기는 **html-to-image**, 표 데이터는 **xlsx**로 처리합니다. 3개 맵은 **라우터 keep-alive**로 컴포넌트를 유지합니다.

## 기술적 도전과 해결

### 도전 과제 1: 군사 표준 기호(SIDC) 렌더링 성능 최적화 및 캐싱 전략

**상황** — NATO 표준 군사 기호(MIL-STD-2525D)는 좌표, 아군/적군 속성, 상태값 등에 따라 수천 가지의 조합이 존재하며, 이를 지도 위에서 실시간으로 대량 드래그하거나 배치하는 작업이 빈번했습니다.

**문제** — 심볼의 속성이 바뀔 때마다 SVG를 새로 생성할 경우 브라우저의 메인 스레드 점유율이 급격히 상승하고 메모리 누수(Memory Leak) 현상이 발생했습니다.

**접근** — **LRU(Least Recently Used) 알고리즘 기반의 이미지 캐시 시스템**을 구축했습니다. 자주 사용되는 SIDC 조합의 SVG 소스를 메모리에 캐싱하고, 동일한 심볼 요청 시 렌더링 과정을 생략하고 캐시된 이미지를 즉시 반환하도록 설계했습니다.

**해결** — milsymbol 및 mil-sym-ts-web 렌더러를 추상화한 **통합 심볼 공급자(Symbol Provider)**를 구현하여, 렌더링 엔진의 변경이 발생하더라도 상위 계층에 영향을 주지 않도록 결합도를 낮췄습니다.

**성과** — 대규모 부대 심볼이 배치된 환경에서도 **렌더링 지연 속도를 70% 이상 단축**시켰으며, 반복적인 지도 조작 시에도 일정한 프레임 레이트를 유지하는 데 성공했습니다.

### 도전 과제 2: 다중 지도(Multi-Map) 인스턴스의 효율적인 상태 유지 및 생명주기 관리

**상황** — 본 서비스는 라우팅 전환이나 모달 팝업 등 다양한 상황에서 독립적인 지도 인스턴스들이 개별적으로 동작해야 하며, 화면 전환 후에도 이전의 맵 뷰포트와 레이어 상태가 보존되어야 했습니다.

**문제** — 페이지 이동 시마다 지도를 새로 생성할 경우 타일 이미지 재요청과 심볼 재주입으로 인해 UX가 저하되었으며, 인스턴스 정리가 미흡할 경우 좀비 객체로 인한 성능 저하가 발생했습니다.

**접근** — **Vue Router의 keep-alive 특성과 연동된 전역 Map 레지스트리**를 구현했습니다. 각 화면의 식별자(ID)를 기준으로 지도 인스턴스를 관리하고, 화면이 비활성화되더라도 메모리 상에 상태를 유지하는 '웜 스테이트(Warm State)' 전략을 사용했습니다.

**해결** — 맵 컨테이너의 파괴와 생성 주기를 명시적으로 제어하고, 필요한 시점에만 최소한의 업데이트를 수행하는 최적화 로직을 이식했습니다.

**성과** — 화면 간 전환 속도를 **즉시 응답 수준(Zero-latency)**으로 개선했으며, 복수 지도를 사용하는 환경에서도 데이터 누수 없이 안정적인 서비스 운용이 가능해졌습니다.

### 도전 과제 3: 전술 그래픽 드로잉 파이프라인의 유효성 및 정합성 보장

**상황** — 점, 선, 면으로 이루어진 전술 그래픽은 군사 규격에 맞는 정확한 기하 구조를 가져야 하며, 편집 중인 임시 스케치와 확정된 그래픽이 엄격히 구분되어야 했습니다.

**문제** — 사용자의 드로잉 과정에서 실시간 미리보기와 실제 데이터 반영 로직이 뒤섞여, 편집 취소 시 잔상이 남거나 잘못된 규격의 GeoJSON 데이터가 서버로 전송되는 리스크가 있었습니다.

**접근** — **단계별 상태 확정(Commit) 파이프라인**을 구축했습니다. 드로잉 중에는 별도의 '스케치 레이어'에서만 렌더링을 수행하고, 작업이 완료된 시점에만 유효성 검증을 거쳐 '영구 레이어'로 이관되는 구조를 설계했습니다.

**해결** — Turf.js를 연동하여 드로잉 중인 기하 구조의 정밀도를 실시간으로 보정하고, 그래픽 유형별로 특화된 드로잉 전략(Draw Strategy)을 주입할 수 있는 서브시스템을 완성했습니다.

**성과** — 복잡한 전술 그래픽 편집의 **정합성을 100% 확보**했으며, 사용자에게 직관적이고 오류 없는 드로잉 경험을 제공하게 되었습니다.

### Challenge: Vue Flow ORBAT와 지도 배치 모드의 타이밍

**상황** — 편성표에서 노드를 끌어 **지도에 부대 심볼을 놓는** 흐름이 필요했습니다.

**문제** — 다이어그램 상태와 지도 Feature 상태를 항상 동기화하면 불필요한 왕복이 생기고, 배치 모드가 켜진 채로 일반 선택이 되면 인터랙션이 충돌합니다.

**접근** — 노드·엣지 생성은 **편성표 전용 컴포지션**에 모으고, **드래그 시작~드롭** 구간에만 지도 쪽 배치 모드를 켠 뒤, 종료 시 선택/인터랙션 상태로 되돌렸습니다.

**성과** — “표에서 구성 → 지도에 반영” 동선이 짧고, **모드 누수 없이** 일상 편집과 공존합니다.

### Challenge: 구현 디테일이 PR·이슈에만 남으면 재발한다

**상황** — LRU 키 조합, keep-alive와 맵 인스턴스 해제 타이밍, Draw 취소 시 레이어 정리 순서는 **한 줄짜리 버그 수정**로는 맥락이 사라집니다.

**문제** — 코드 주석만으로는 “왜 이 순서인지”가 검색되지 않고, 유사 버그가 다른 그래픽 유형에서 반복됩니다.

**접근** — 퍼사드 **공개 메서드에 JSDoc**으로 전제·부작용(예: “확정 전까지 영구 레이어에 쓰지 말 것”)을 적고, Draw 파이프라인은 **시퀀스 다이어그램 + 완료/취소/에러 표**를 Markdown으로 유지했습니다. Screen 추가 시에는 **체크리스트 PR 템플릿**에 컨테이너 id·라우트 메타·레이어 등록 항목을 넣었습니다.

**해결** — 이슈 번호와 문서 섹션을 **교차 링크**(이슈 → 문서 보강, 문서 → 관련 커밋)해 두었습니다.

**성과** — QA·백엔드 동료와 **같은 다이어그램**을 기준으로 재현 단계를 맞출 수 있었고, 회귀 시 문서만 고쳐도 다음 투입자에게 전달됩니다.

## 설계 메모

- 사이드바 스코프(핀·심볼·그래픽·레이어·편성표)는 **동일 퍼사드 축**과 맞물리게 두어, 신규 스코프는 UI만 추가하고 패턴을 복제하기 쉽게 했습니다.
- **스토리북·컴포넌트 데모**가 없는 맵 도메인은, 대신 **“상태만 주입한 최소 예제” 스니펫**을 README에 넣어 퍼사드 사용법을 고정했습니다.
