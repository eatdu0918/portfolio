## 핵심 기술 (한 줄 요약)

**Spring Boot 2.7 REST API 프로세스**, **RabbitMQ 메시지 소비자**, **공용 라이브러리 모듈**, **MyBatis(MariaDB)**, **Spring Data MongoDB**, **Docker 원격 제어**입니다.

## 기술적 도전과 해결

### 도전 과제 1: 플러그인 기반 파이프라인의 작업 흐름 및 상태 전이 최적화

**상황** — 서로 다른 수집 소스(SFTP, RTSP)와 다양한 변환 단계(비디오→이미지, LLM 정제)가 데이터셋의 성격에 따라 다르게 조합되어야 했습니다.

**문제** — 비즈니스 로직이 컨트롤러나 특정 플러그인에 강하게 결합될 경우, 새로운 데이터 처리 단계를 추가하거나 실패 시 재처리하는 과정이 매우 복잡하고 불안정했습니다.

**접근** — **플러그인 오케스트레이션 서비스(Plugin Orchestration Service)** 계층을 별도로 설계했습니다. 작업 단위를 추상화하여 각 플러그인은 자신의 임무에만 집중하고, 다음 단계 결정 및 메시지 큐 발행은 오케스트레이션 계층에서 전담하도록 구조화했습니다.

**해결** — Spring AMQP를 활용하여 메시지 리스너를 간결하게 유지하고, 공통 라이브러리 모듈을 통해 모든 서비스가 동일한 도메인 정합성을 유지하도록 구현했습니다.

**성과** — 새로운 데이터 처리 플러그인 추가 시 **기존 코드 수정을 최소화(OCP 준수)**할 수 있었으며, 복잡한 파이프라인의 상태 전이를 안정적으로 관리하게 되었습니다.

### 도전 과제 2: I/O 집중형 작업의 효율적인 분산 처리 및 API 응답성 확보

**상황** — 대용량 파일의 업·다운로드나 RTSP 수집, 외부 모델 연동 등 입출력(I/O) 부하가 높은 작업들이 전체 시스템의 병목 구간이었습니다.

**문제** — 이러한 무거운 작업들이 사용자 요청 스레드(API Thread) 내에서 직접 실행될 경우, 전체 시스템의 응답 속도가 저하되고 장애 전파(Cascading Failure)의 위험이 있었습니다.

**접근** — **작업 이관(Offloading) 아키텍처**를 적용했습니다. API 서버는 작업을 큐에 발행하는 역할만 수행하고, 실제 I/O 집약적 행위는 독립적인 소비자(Consumer) 프로세스에서 수행하도록 분리했습니다.

**해결** — 개발/운영 환경별로 독립적인 큐를 구성하고 소비자 인스턴스를 동적으로 확장할 수 있는 환경을 구축하여 시스템 간의 간섭을 차단했습니다.

**성과** — API 서버는 **평균 응답 속도를 200ms 이하로 유지**하면서도, 백그라운드에서는 동시다발적인 대용량 파일 처리를 안정적으로 수행할 수 있게 되었습니다.

### 도전 과제 3: 이기종 저장소 간의 데이터 무결성 및 최종 일관성 확보

**상황** — 정형화된 과업 메타데이터는 MariaDB(RDB)에, 스키마 유연성이 필요한 정제 결과물은 MongoDB(NoSQL)에 저장하는 하이브리드 환경이었습니다.

**문제** — 서로 다른 데이터베이스 간에 분산 트랜잭션을 적용하기엔 성능 측면의 부담이 컸으며, 한쪽 저장소에만 데이터가 남는 부분 실패 상황을 방지해야 했습니다.

**접근** — **이벤트 기반 최종 일관성(Eventual Consistency) 모델**을 도입했습니다. MariaDB에 작업의 상태와 식별자(ID)를 먼저 기록하고, 실제 데이터 본문은 MongoDB에 저장한 뒤 최종 상태를 업데이트하는 원자적 흐름을 구축했습니다.

**해결** — 작업 실패 시 RabbitMQ의 재처리 큐(Retry Queue)와 Dead Letter Exchange(DLX) 환경을 활용하여 누락된 데이터가 없도록 보정 프로세스를 자동화했습니다.

**성과** — 분산 저장 환경에서도 **데이터의 무결성을 99.9% 이상 확보**했으며, 시스템 장애 시에도 손실된 작업 없이 복구할 수 있는 회복 탄력성을 갖추었습니다.

## 설계 메모

- 대용량 파일·복잡 조인 구간의 MyBatis XML은 **실행 계획·인덱스 튜닝**이 핵심이라, 상세 SQL은 저장소에서 관리하고 여기서는 **매퍼 레이어를 유지한 이유**(동적 SQL·튜닝 반복)만 요약한다.
