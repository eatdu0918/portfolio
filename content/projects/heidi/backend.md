## 핵심 기술 (한 줄 요약)

**Spring Boot 3.2 · Maven 멀티모듈(사용자 API·관리 백오피스·공통 모듈) · MariaDB · MyBatis(+일부 JPA) · Spring Security(JWT) · RabbitMQ · Redis · 실시간 웹소켓 메시징**입니다.

## 기술적 도전과 해결

### Challenge: 사용자 API와 백오피스 API의 배포·공격면 분리

**상황** — 운영자 기능과 고객 기능은 **포트·권한·트래픽 패턴**이 다릅니다.

**문제** — 한 애플리케이션에 몰면 실시간 채널까지 포함한 사용자 API가 관리 기능과 섞입니다.

**접근** — **실행 가능한 JAR/WAR 두 개**와 **공통 라이브러리**로 나눴습니다.

**해결** — 사용자 API는 **실시간 구독**까지 포함, 백오피스는 REST 위주로 공격면을 줄였습니다.

**성과** — 인프라에서 **트래픽·스케일 정책을 분리**할 수 있었습니다.

### Challenge: SaaS 테넌트 경계와 인증 일관성

**상황** — 모든 요청에서 회사(테넌트)와 사용자 신원이 일치해야 합니다.

**문제** — 토큰에 테넌트 정보가 없으면 후속 쿼리에서 누수가 납니다.

**접근** — JWT 클레임에 **로그인 식별자·회사·처리 큐 힌트**를 넣어 REST·실시간 메시징·업로드가 같은 맥락을 쓰게 했습니다.

**해결** — BCrypt·필드 암호화·필터 체인을 한 축으로 유지했습니다.

**성과** — 중복 로그인·강제 로그아웃 같은 **운영 정책을 한곳**에서 적용할 수 있었습니다.

### Challenge: 복잡 조회는 SQL, 일부 도메인은 JPA

**상황** — 리포트·목록 쿼리는 튜닝이 잦고, 일부 엔티티는 객체 모델이 편했습니다.

**문제** — JPA만 쓰면 연쇄 조회·동적 필터가 힘듭니다.

**접근** — **MyBatis를 주력**, JPA는 제한적으로 병행했습니다.

**해결** — HikariCP 풀과 MariaDB 드라이버로 연결 안정성을 맞췄습니다.

**성과** — 성능 이슈가 나도 **SQL을 직접 다룰 수 있는** 운영 친화 구조를 유지했습니다.

### Challenge: 메시징·캐시·실시간의 역할 분담

**상황** — 작업 생성 후 긴 파이프라인이 이어집니다.

**문제** — 모든 이벤트를 DB에 쓰면 병목이 됩니다.

**접근** — RabbitMQ로 **작업을 넘기고**, Redis로 **진행 이벤트를 흡수**, MariaDB에는 **최종 상태**를 남깁니다.

**해결** — 브라우저 실시간 채널은 API의 **단일 구독 지점**에서만 열었습니다.

**성과** — Python 워커를 늘려도 **API 스키마를 자주 바꾸지 않아도** 되었습니다.

## 설계 메모
