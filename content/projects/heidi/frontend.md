## 핵심 기술 (한 줄 요약)

**Vue 3 + Vite + Electron 30** 데스크톱(PrimeVue·Tailwind·Pinia·다국어 모듈)과 **Vue 3 웹 백오피스**를 분리했고, 실시간은 **웹소켓 기반 메시징**, 대용량 파일은 **사전 로드 브리지 + SFTP**로 처리합니다.

## 기술적 도전과 해결

### Challenge: 브라우저만으로는 부족한 대용량·폴더 단위 업로드

**상황** — 현장 사용자는 폴더·SFTP·로컬 캐시가 필요했습니다.

**문제** — 순수 브라우저만으로는 파일 시스템·장시간 전송이 불안정합니다.

**접근** — **Electron 메인 프로세스**에서 SFTP·SQLite·FFmpeg를 돌리고, **사전 로드 스크립트와 안전한 브리지**로 렌더러에 노출하는 API를 최소화했습니다.

**해결** — 작업 생성 → SFTP 업로드 → 수동 편집 중 SQLite 임시 저장 → 서버 동기화 흐름으로 정리했습니다.

**성과** — 권한이 필요한 작업을 **메인에 가두어** 보안·안정성을 동시에 맞췄습니다.

### Challenge: 장시간 작업의 진행률·완료 신뢰도

**상황** — AI 단계가 길어 사용자는 “멈춘 것 같다”는 불안을 느낍니다.

**문제** — 폴링은 서버·클라이언트 모두 비용이 크고, 이벤트가 잘리면 상태가 어긋납니다.

**접근** — 실시간 구독 후 메시지 유형을 보고 **앱 내부 이벤트 허브**로 화면에 전달했습니다.

**해결** — 진행 스토어를 웹소켓과 맞물리게 설계하고 필요한 값만 **브라우저 로컬 저장소**에 남겼습니다.

**성과** — 탭 이동·재접속 후에도 **진행 UI가 끊기지 않게** 했습니다.

### Challenge: 수동 비식별(캔버스)과 비디오 제로샷 트래킹

**상황** — 자동 탐지가 빗나간 구간만 사람이 고치고 싶었습니다.

**문제** — 이미지와 비디오의 편집 모델이 달라 UX가 갈라집니다.

**접근** — 이미지는 영역 선택 후 로컬+API, 비디오는 초기 박스 → 트래킹 결과 → 전 프레임 블러로 **단계형**으로 나눴습니다.

**해결** — 서버 파이프라인(**제로샷 트래킹 단계 → 구간 블러 단계**)과 화면 상태를 1:1로 맞췄습니다.

**성과** — “완전 자동”과 “사람 보정”을 **한 제품 안**에서 전환할 수 있었습니다.

### Challenge: 백오피스는 웹, 작업은 Electron으로 나눈 이유

**상황** — 관리자는 설치 없이 브라우저에서 테넌트·공지·감시를 해야 했습니다.

**문제** — Electron까지 요구하면 운영 비용이 큽니다.

**접근** — **백오피스는 경량 Vue SPA**, 실제 비식별 작업은 Electron으로 분리했습니다.

**해결** — 공지 WYSIWYG·다국어 첨부는 웹에 집중, SFTP·로컬 DB는 데스크톱에만 두었습니다.

**성과** — 역할별 **배포 형태를 최적화**했습니다.

## 설계 메모

- 다국어 리소스는 운영·현장 모두에서 동일한 제품 언어 경험을 주기 위해 **문자열 키 설계를 일찍 고정**했습니다.
